<objective>
Build `IssueFilingRunner` â€” the orchestration layer above `CLIRunner` that turns a transcript +
bound repo + provider config into a filed issue. It acquires the GitHub token, writes a
per-invocation MCP config tempfile, assembles the scoped `claude -p` command, runs it via the
extended `CLIRunner` with the token passed through the environment and cwd = the bound repo, and
maps the result through `IssueResultParser`.

Purpose: This is the single place that enforces the locked spike constraints (scoped tool grant,
structured `stream-json --verbose` output, env-var token passthrough, POSIX-escaped transcript,
absolute `--mcp-config` path, `--strict-mcp-config`). AppState (04-03) calls it through a seam.

Output: `IssueFilingRunner.swift` and its tests. The Runner is fully unit-testable for command
assembly, prompt construction, and MCP-config JSON without spawning a real `claude` process; the
end-to-end real-filing check is the human-verify checkpoint in 04-04.
</objective>

<accepted_v1_behavior>
## Accepted v1 Behavior â€” Auto-File, No Review Screen (LOCKED by user)

v1 invokes the AI CLI once (single-shot) and AUTO-FILES the issue through the CLI's MCP server.
There is NO confirm-before-file step. The CLI investigates the repo and MAY file an issue that
diverges from the literal transcript because it corrects false premises (spike 002: the transcript
claimed the README lacked a testing section; the model found one and filed an accurate "improve the
docs" issue instead). This accuracy-vs-fidelity tension is an explicitly ACCEPTED v1 behavior â€” the
prompt instructs the model to "file it directly. Do not ask for confirmation." A confirmation gate
is deferred to v2 (REVIEW-01). The executor MUST NOT add any review/confirm step.
</accepted_v1_behavior>

<assumptions>
## Explicit Assumptions (surfaced from research Open Questions)

- **Owner/repo discovery (Open Q1):** v1 passes `ownerRepo: nil` from the AppState seam (04-03), so
  the prompt names "the repository in the current working directory" and the model infers owner/repo
  from cwd `.git/config`. `git remote get-url origin` parsing is deferred to v1.1 if model accuracy
  proves insufficient. The `ownerRepo` parameter exists so that upgrade needs no signature change.
- **Token source (Open Q3 / AUTH-01):** env-var-first (`GITHUB_PERSONAL_ACCESS_TOKEN` in the
  process environment), falling back to `gh auth token`. This makes `gh` optional for users who set
  the env var directly.
- **Docker + claude are runtime prerequisites** (RESEARCH Environment Availability). v1 does not
  bundle them; absence surfaces as a `permissionDenied`/`cliFailed` error, not a crash.
</assumptions>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| transcript (untrusted text) â†’ `claude -p` shell command | The spoken transcript is embedded into the `/bin/zsh -lc` command string |
| app â†’ AI CLI subprocess (cwd = user's repo) | The app shells out to the user's AI CLI with cwd set to their real repository |
| GitHub token â†’ subprocess environment | The PAT crosses into the subprocess (and onward to the Docker MCP server) |

| T-04-04 | Tampering | transcript embedded in `claude -p` command string | mitigate | POSIX single-quote escape (`shellEscape`, reused from `Transcriber.prepare`) so the transcript is one shell word; command injection via quotes/`;`/`$()` is neutralized. |
| T-04-05 | Information Disclosure | GitHub PAT | mitigate | Token passed via `Process.environment` (`environment:` param), never in the `-lc` command string; never logged; never written to the MCP config file (Docker reads it from the inherited env via `-e GITHUB_PERSONAL_ACCESS_TOKEN` with no value). |
