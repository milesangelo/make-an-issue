  <what-built>
    The complete v1 pipeline: push-to-talk capture (Phase 2) → bundled-whisper transcription
    (Phase 3) → the AI CLI files a real GitHub issue via its MCP server with cwd = the bound repo
    (04-02) → the app parses the issue number from the url and speaks "created issue #NUMBER"

  <how-to-verify>
    1. From a terminal inside a real GitHub repo you can file issues in (and that is `gh auth`'d),
       run the launcher: `bin/make-an-issue` (or the documented launch command). Confirm the
       menu-bar icon appears and the menu shows that repo as the bound repository.
    2. (Optional) Open the menu and confirm the "CLI Command" field shows `claude`. Leave it as is.
    3. Hold the push-to-talk shortcut (default Control-Option-I), speak a short, clear feature/bug
       idea about THIS repo (e.g. "the README should document how to run the test suite"), then
       release.
    4. Watch the menu "Recording" line move through Recording… → Transcribing… → Filing issue….
       Filing typically takes ~30s–2min (repo investigation + MCP filing).
    5. CONFIRM you HEAR the app speak "created issue #NUMBER" with a small human-facing number
       (e.g. "#90"), NOT a 10-digit number and NOT a base64 string. (If it speaks a huge number, the
       parser is reading the id field — that is a defect to fix in this plan.)
    6. Open GitHub (or run `gh issue view <NUMBER>`) and CONFIRM a real issue with that exact number
       exists in the bound repo, created just now. Note that the issue body may differ from your
       literal words — the CLI investigates the repo and may correct false premises. This divergence
       is the ACCEPTED v1 behavior; it is NOT a defect.
    7. NEGATIVE CHECK (no false success): with Docker stopped (or `gh` signed out), repeat steps 1–4.
       CONFIRM the app does NOT speak a success — it shows a status message (e.g. "Docker required…"
       / "Sign in to GitHub first…") and returns to Idle. Restore Docker / `gh auth` afterward.
  </how-to-verify>
