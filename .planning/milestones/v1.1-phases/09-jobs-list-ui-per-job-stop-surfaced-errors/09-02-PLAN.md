<objective>
Render the existing `AppState.jobs` model as a live "Filing Jobs" section in the left-click popover, wiring the Stop / ✕ / Clear-all controls to the surface built in Plan 09-01. This is the visible payload of JOBS-01, JOBS-02, and RESIL-01.

The section (D-11) sits at the bottom of `MenuView.body` after the existing `TranscriptCard`, is hidden entirely when there are no jobs (D-12), scrolls inside a fixed max-height area with newest job on top (D-03), and renders one `JobRow` per job over a single `ForEach(appState.jobs.reversed())` (D-01). Every job state gets a row: `.filing` (spinner + Stop, JOBS-02/D-05), `.done` (clickable `#N` → issue URL, D-10), `.failed` (mapped message + expandable transcript, RESIL-01/D-08), `.cancelled` (secondary "Filing cancelled"). Control separation is strict per D-05: active rows get Stop only, terminal rows get ✕ only, Clear-all removes terminal rows only.

Purpose: Turn the already-solved backend (jobs model + cancel path from Phases 5-6, plus Plan 09-01's dismiss/clearFinished/message/JobRowStyle) into the user-visible jobs surface — the last piece of the v1.1 milestone.

Output: `JobsSection`, `JobRow`, `DismissButton`, `ClearAllButton`, `TranscriptSnippet` in `MenuView.swift`, with `JobsSection` composed into `MenuView.body`.

Prohibitions (locked): NO Retry button (D-07 — dismiss-only); NO ✕ on `.filing` rows and NO Stop on terminal rows (D-05); NO direct mutation of `appState.jobs` from a view (route through `cancel`/`dismiss`/`clearFinished`); do NOT re-author error strings (use `AppState.message(for:)`, D-09); do NOT reuse `WaveformView` for the filing indicator (wrong metaphor — use `ActivitySpinner`); do NOT add a `createdAt` field to `FilingJob` (`.reversed()` is sufficient, Pitfall 3).
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| AI-CLI subprocess stdout → app → OS | The done-row opens `job.result.url` (regex-parsed from `claude` stdout) via `NSWorkspace.shared.open` |
| UI controls → AppState | Stop/✕/Clear-all buttons invoke `cancel`/`dismiss`/`clearFinished`; views never mutate `jobs[]` directly |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-03 | Tampering / Spoofing | done-row `NSWorkspace.shared.open` call site (`JobRow`, MenuView.swift) | mitigate | Open only the `URL` returned by `JobRowStyle.openableIssueURL` (https-validated + unit-tested in Plan 09-01); no-op when it returns nil. Never call `open` on a raw `result.url` string (ASVS V5) |
| T-09-04 | Information Disclosure | rendering `AppState.message(for: job.error)` + `job.transcript` in `Text` (`JobRow`/`TranscriptSnippet`) | accept | Strings are local user-generated (transcript) or from the closed `IssueFilingError` enum's fixed messages; SwiftUI `Text` auto-escapes and no `Text(markdown:)`/HTML interpolation is used — no injection surface, low severity |
| T-09-05 | Tampering | UI-reachable `✕`/`Clear-all` → `dismiss`/`clearFinished` (loss of in-flight work) | mitigate | Buttons call the terminal-only methods from Plan 09-01 (enforced + tested); `.filing` rows expose Stop only, never ✕ (D-05), so no active job can be dismissed without an explicit Stop first |
| T-09-SC | Tampering | npm/pip/cargo/SPM installs | accept | No package-manager installs in this phase; `Package.swift` unchanged (verified). No supply-chain surface introduced |
</threat_model>
