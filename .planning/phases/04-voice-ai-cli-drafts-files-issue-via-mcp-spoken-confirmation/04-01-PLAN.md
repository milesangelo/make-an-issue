<objective>
Build the three independent, dependency-free foundation pieces for AI-CLI issue filing:
the stdout parser (`IssueResultParser`), the provider seam (`IssueFilingConfig`), and the
`CLIRunner` environment-passthrough parameter. Each is pure or purely additive, has no
dependency on the others, and ships with unit tests.

Purpose: These three artifacts are consumed by `IssueFilingRunner` (plan 04-02). Defining them
first — with their types, error enums, and the `IssueFilingResult` contract — means the Runner
plan implements against fixed contracts instead of discovering them.

Output: `IssueResultParser.swift`, `IssueFilingConfig.swift`, the extended `CLIRunner.swift`,
and three test files. All existing tests stay green; this plan only adds new symbols and one
optional parameter with a default.

  <name>Task 1: IssueResultParser — JSONL walk + prose fallback + permission_denials gate</name>
  <files>Sources/MakeAnIssue/IssueResultParser.swift, Tests/MakeAnIssueTests/IssueResultParserTests.swift</files>
  <read_first>
    - .claude/skills/spike-findings-make-an-issue/sources/002-claude-github-file-issue/parse-issue.js (the two-pass algorithm to port)
    - .claude/skills/spike-findings-make-an-issue/references/github-issue-filing.md (url-not-id rule)
    - .planning/phases/04-voice-ai-cli-drafts-files-issue-via-mcp-spoken-confirmation/04-RESEARCH.md (Pattern 3 — JSON shapes; Pitfall 3 and 4)
    - Tests/MakeAnIssueTests/TranscriberTests.swift (XCTAssertThrowsError + MARK-section test style)
  </read_first>
  <behavior>
    - Given a JSONL stdout whose assistant message carries a tool_result block with content
      `{"id":"<node-id>","url":"https://github.com/owner/repo/issues/89"}`, parse() returns
      IssueFilingResult(number: 89, url: ".../issues/89"). The number comes from the url path,
      NOT the id field.
    - Given a JSONL stdout with no tool_result block but a final result event whose result string
      contains "Issue URL: https://github.com/owner/repo/issues/42", parse() returns number 42
      via the prose fallback.
    - Given a result event whose permission_denials array is non-empty, parse() throws
      IssueParseError.permissionDenied with the denied tool names — even when the rest of the
      stream looks successful (no issue was filed).
    - Given a stream with neither a parseable url nor a prose url, parse() throws
      IssueParseError.noIssueFound.
    - When both a structured tool_result url and a prose url are present, the structured one wins.
  </behavior>
  <action>
    Create `IssueResultParser.swift` (import Foundation). Declare `struct IssueFilingResult` with
    `let number: Int` and `let url: String`, and `enum IssueParseError: Error, Equatable` with
    cases `permissionDenied([String])`, `noIssueFound`, `malformedOutput` — matching the typed-enum
    style of `TranscriberError`. Implement `static func parse(stdout: String) throws ->
    IssueFilingResult` following the algorithm in `04-RESEARCH.md` Pattern 3 (which is a faithful
    Swift port of the reference parser in the read_first JS source).

    Split stdout on newlines; for each line, JSONSerialization-decode it to `[String: Any]`,
    skipping lines that fail to decode (the stream is newline-delimited complete objects). On a
    `type == "result"` object, capture the `result` string and read `permission_denials` as
    `[[String: Any]]`, collecting each entry's `tool_name`. On a `type == "assistant"` object, walk
    `message.content` for blocks whose `type == "tool_result"`, normalize the block `content`
    (String, or array-of-objects joined on their `text`), and extract the issue url from it.

    Declare the regexes as `static let` NSRegularExpression properties initialized once. Use these
    patterns: structured `"url"\s*:\s*"(https?://github\.com/[^"]+/issues/(\d+))"`; prose
    `https?://github\.com/[^\s)"']+/issues/(\d+)`. In both cases the issue number is the trailing
    `(\d+)` capture group from the url path. Per the spike findings, the issue number must be read
    from the url path and never from any id field in the result JSON.

    Resolution order in parse(): if `permission_denials` is non-empty, throw
    `.permissionDenied(deniedTools)` first (a denied tool means nothing was filed — do not trust
    exit success). Otherwise return the structured tool_result result if found; else the prose
    fallback if found; else throw `.noIssueFound`.

    Write `IssueResultParserTests.swift` (import XCTest, @testable import MakeAnIssue) mirroring the
    `TranscriberTests` structure: MARK sections per parse path, inline `let stdout = """..."""`
    JSONL fixtures (no file I/O). Cover every bullet in <behavior>, including an assertion that the
    spoken-worthy number is 89 (not the 10-digit node id) for the structured case, and an
    XCTAssertThrowsError check that the denied-tools case throws `.permissionDenied`.
  </action>
  <verify>
    <automated>swift test --filter IssueResultParserTests</automated>
  </verify>
  <done>IssueResultParser.parse extracts number from url (never id), falls back to prose, gates on permission_denials, and throws noIssueFound when nothing parses; all IssueResultParserTests pass.</done>
</task>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller → CLIRunner subprocess | The GitHub token crosses into the subprocess environment here |
| AI CLI stdout → IssueResultParser | Untrusted (model-generated) text is parsed for the issue url |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Information Disclosure | token passed to subprocess | mitigate | `CLIRunner.environment` merges the token into `Process.environment`, NOT the `-lc` command string — keeps it out of `ps` output (Pitfall 2). |
| T-04-02 | Tampering | parsing AI-CLI stdout | mitigate | Parser gates on a non-empty `permission_denials` array and treats it as failure even on apparent success; never trusts model prose alone for filing confirmation. |
| T-04-03 | Spoofing | wrong number spoken | mitigate | Parser extracts the number from the url `/issues/(\d+)` path only, never from the `id` node-id field (spike 002 landmine). |
</threat_model>
