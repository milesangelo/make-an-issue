<objective>
Implement the core process-tree termination mechanism that CANCEL-01/02 depend on, without
disturbing the single-resume `RunState` invariant (SC-4).

Three coupled changes, interface-first:
1. **CLIRunner group kill** — replace the single-PID kills on the timeout escalation path with
   process-group kills (negated identifier) so the whole `zsh → claude → docker run` tree receives
   the signal, keeping SIGTERM-before-SIGKILL ordering with the existing 2s grace.
2. **CLIRunner cancellation bridge + pgid hook** — wrap the existing `withCheckedContinuation` in
   `withTaskCancellationHandler` so `Task.cancel()` drives a group SIGTERM → 2s grace → group SIGKILL
   (onCancel sends signals only; terminationHandler stays the sole resume path), and surface the
   spawned process group id through a new optional `onSpawn` callback.
3. **IssueFilingRunner seam + FilingJob field** — forward the pgid via a new
   `onProcessStarted` callback, raise `CancellationError` with `try Task.checkCancellation()` after
   `CLIRunner.run` returns (the existing `defer` still deletes the MCP tempfile on the cancel path),
   and add `FilingJob.processGroupID: pid_t?` so the quit path (06-04) can force-kill survivors.

Purpose: CANCEL-01 (kill the full tree, SIGTERM-first) and CANCEL-02 (surface cancellation) at the
subprocess layer; SC-4 (no double-resume) preserved.
Output: a CLIRunner that reaps the whole tree on cancel/timeout, a CancellationError-aware
IssueFilingRunner, the `processGroupID` handle, and the two CLIRunner cancel tests passing.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Swift Task / @MainActor AppState → spawned `zsh → claude → docker run --rm` process tree | The cancel signal crosses from the in-process Swift world into a real OS process group via POSIX `kill(-pgid, …)`; the negated identifier is the only thing keeping the signal inside the child tree. |
| CLIRunner continuation ↔ concurrent completion paths (terminationHandler, timeout Task, onCancel) | Multiple closures can fire near-simultaneously; the risk is a double-resume of the CheckedContinuation. |
