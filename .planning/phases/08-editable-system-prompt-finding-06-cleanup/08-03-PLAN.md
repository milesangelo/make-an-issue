<objective>
Build the editable-instructions UI (SETTINGS-02, SETTINGS-03). Reorganize `SettingsView` into a
`TabView` with a "Shortcut" tab (the existing push-to-talk `KeyboardShortcuts.Recorder`, unchanged)
and a new "Instructions" tab (D-09) containing: a multi-line `TextEditor` bound to
`@AppStorage(AppState.instructionsKey)` defaulting to `IssueFilingConfig.defaultInstructions`
(persists across launches — SETTINGS-02); a "Reset to Default" `Button` that writes
`IssueFilingConfig.defaultInstructions` back into the field so the user visibly sees the reverted
value (SETTINGS-03, D-07); and a greyed-out, read-only display of the always-appended enforced
contract — `IssueFilingRunner.enforcedTrailer` plus the scoped `--allowedTools` grant from
`IssueFilingConfig.claudeGitHub.allowedToolsArgument` — so the user understands the URL-line +
tool-scope contract is always added and cannot be broken by edits (D-04). The window stays fixed-size
with a fixed-min-height editor (D-10).

Purpose: give the developer a transparent, persisted control over drafting guidance while making the
unbreakable contract visible.
Output: a two-tab Settings window, a persisted editable instructions field, a visible Reset, and a
read-only enforced-contract display — all reading the real source-of-truth constants so the display
cannot drift.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user input in TextEditor → @AppStorage(instructionsKey) | Untrusted free-text is persisted to UserDefaults; it is consumed only as prompt guidance (enforcement lives in Plan 02's `buildPrompt`). |
| enforced-contract display → user | Read-only surface that must truthfully reflect what the app actually appends. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-05 | Tampering | Persisted instructions consumed downstream by the runner | mitigate | Architectural: the view only writes the guidance string to `@AppStorage(instructionsKey)`; it has no path to the `--allowedTools` flag or the enforced trailer. Enforcement is owned by `buildPrompt`/`assembleCommand` (Plan 02). The UI cannot bypass the contract. |
| T-08-06 | Information Disclosure (misleading UI) | Read-only enforced-trailer display drifting from the real appended contract | mitigate | The display renders `IssueFilingRunner.enforcedTrailer` and `IssueFilingConfig.claudeGitHub.allowedToolsArgument` directly — no hardcoded/paraphrased copy — so it cannot claim a contract the app does not actually apply. Verified by source assertion in Task 1 and the human-verify checkpoint. |
| T-08-10 | Denial of Service | User clears the field entirely, expecting empty guidance | mitigate | Handled upstream by D-08 (Plan 02): a blank field falls back to `defaultInstructions` at prompt-build time, so filing stays robust regardless of the persisted value. |
| T-08-SC | Tampering | npm/pip/cargo installs | accept | No package installs this phase — `Package.swift` unchanged; no new dependency added. No `[ASSUMED]`/`[SUS]` packages. |
</threat_model>
